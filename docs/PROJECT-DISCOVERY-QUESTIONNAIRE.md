# Project Discovery Questionnaire

## Purpose

Use this questionnaire when you have a project idea but do not yet understand
it well enough to create a roadmap, milestones, or implementation tasks.

This is the layer between:

- raw idea capture
and
- structured project planning

The goal is to turn a high-level idea into a clearer product definition.

## How To Use This

1. Start with the raw idea in simple language.
2. Answer the questions below without trying to design the whole system yet.
3. Turn the answers into a short product brief.
4. Use that product brief to create roadmap and milestone plans.

Do not worry about having perfect answers on the first pass.
The purpose is clarification, not premature certainty.

## Section 1: The Raw Idea

Answer in plain language:

- What is the idea?
- What made you think of it?
- What are you hoping it can do?

This is the rough starting point, not the final project definition.

## Section 2: Problem And Value

Answer:

- What problem does this solve?
- What frustration, pain, or inefficiency exists today?
- Why does this matter?
- What value does the user get if this works?

If you cannot explain the value clearly, the idea may still be too vague.

## Section 3: User And Context

Answer:

- Who is the primary user?
- What kind of person or team is this for?
- In what situation would they use it?
- Who is not the target user?

This helps prevent building for everyone and no one.

## Section 4: Core Outcome

Answer:

- After using this, what should be true for the user?
- What is the single most important result the product should deliver?
- If the user only gets one benefit, what should it be?

This helps define the center of the product.

## Section 5: Main User Flows

List the main things a user needs to do.

Examples:

- create something
- review something
- search something
- save something
- share something
- export something

Then answer:

- What is the primary workflow from start to finish?
- What would the user do first?
- What would happen next?
- What would the user expect at the end?

This is often where the product starts becoming clearer.

## Section 6: Inputs, Outputs, And Data

Answer:

- What information goes into the system?
- What information comes out of the system?
- What needs to be stored?
- What can be temporary?
- What data matters most?

This helps define whether the project is actually a web app, an API, a CLI, a
service, or something else.

## Section 7: Delivery Form

Answer:

- Should this be a web app, mobile app, desktop app, CLI, API, or background service?
- Why is that the right form?
- What environment does it need to run in?
- Does it need to work on a phone, browser, or laptop?

Do not assume "web app" automatically.
Choose the form that fits the user and workflow.

## Section 8: MVP Scope

Define the minimum useful version.

Answer:

- What absolutely must exist for version 1?
- What would make it useful, even if basic?
- What can wait until later?
- What should explicitly not be part of the MVP?

This becomes the first major scope control step.

## Section 9: Constraints

Answer:

- Is this local-only or cloud-based?
- Single-user or multi-user?
- Does it require authentication?
- Are there privacy or compliance concerns?
- Are there budget limits?
- Are there technology constraints?
- Are there time constraints?

Constraints shape the architecture early.

## Section 10: Risks And Unknowns

Answer:

- What parts are technically uncertain?
- What assumptions might be wrong?
- What dependencies could block progress?
- What could make this project harder than it first appears?

This section often reveals where spikes or research tasks are needed.

## Section 11: Success Criteria

Answer:

- How will you know the MVP is working?
- What would count as "good enough to use"?
- What would prove the idea is worth continuing?

This helps define whether the project is succeeding or just accumulating code.

## Section 12: Product Brief Output

After answering the questionnaire, condense the results into a short product
brief with these sections:

- product goal
- target user
- problem statement
- core outcome
- primary user workflow
- MVP scope
- non-goals
- constraints
- major risks
- success criteria

This brief should be short enough to read quickly but complete enough to guide
roadmap creation.

## Quick Screening Questions

If you are evaluating whether an idea is worth pursuing at all, ask:

- Does this solve a real problem?
- Can I explain the value simply?
- Do I know who it is for?
- Can I describe the MVP?
- Is the effort reasonable for the likely value?
- Am I excited enough to work on it after the novelty wears off?

If several answers are weak, keep the idea in the idea list but do not force it
into execution yet.

## Suggested Workflow With An Idea Inbox

A strong workflow is:

1. capture raw ideas quickly in an idea inbox
2. review the inbox later when you have focus
3. select one promising idea
4. run it through this questionnaire
5. create a product brief
6. then use the startup checklist and roadmap process

This keeps brain dumps and real project planning separate, which is healthier
than trying to plan every idea immediately.

## Rule Of Thumb

Do not ask:

- "How do I build this?"

until you can answer:

- "What exactly am I trying to build, for whom, and why?"

That is the purpose of project discovery.
